# SEO.ai Phase 3A Data Model

## New Tables

## `seo_recommendation_runs`
Purpose:
- stores deterministic recommendation generation runs
- captures lineage to persisted audit/comparison evidence
- stores aggregate run-level counts and lifecycle metadata

Core fields:
- `id`
- `business_id`
- `site_id`
- `audit_run_id` (nullable)
- `comparison_run_id` (nullable)
- `status` (`queued` | `running` | `completed` | `failed`)
- `total_recommendations`
- `critical_recommendations`
- `warning_recommendations`
- `info_recommendations`
- `category_counts_json`
- `effort_bucket_counts_json`
- `started_at`
- `completed_at`
- `duration_ms`
- `error_summary`
- `created_by_principal_id`
- `created_at`
- `updated_at`

Integrity:
- check constraint requires at least one lineage input:
  - `audit_run_id IS NOT NULL OR comparison_run_id IS NOT NULL`
- foreign keys to `businesses`, `seo_sites`, `seo_audit_runs`, `seo_competitor_comparison_runs`

Indexes:
- business/site/time listing
- business/status filtering
- business/audit run lookup
- business/comparison run lookup

## `seo_recommendations`
Purpose:
- stores deterministic recommendation items generated by a run
- keeps recommendation records stable for API/report retrieval

Core fields:
- `id`
- `business_id`
- `site_id`
- `recommendation_run_id`
- `audit_run_id` (nullable)
- `comparison_run_id` (nullable)
- `rule_key`
- `category` (`SEO` | `CONTENT` | `STRUCTURE` | `TECHNICAL`)
- `severity` (`INFO` | `WARNING` | `CRITICAL`)
- `title`
- `rationale`
- `priority_score`
- `effort_bucket` (`LOW` | `MEDIUM` | `HIGH`)
- `evidence_json`
- `created_at`
- `updated_at`

Integrity:
- unique constraint to prevent duplicated deterministic rules per run:
  - `(business_id, recommendation_run_id, rule_key)`
- foreign keys to `businesses`, `seo_sites`, `seo_recommendation_runs`, `seo_audit_runs`, `seo_competitor_comparison_runs`

Indexes:
- business/site/time listing
- business/run/priority sorting
- business category/severity/effort filtering
- run and lineage lookups

## Scoping and Lineage
All recommendation tables are business-scoped and site-scoped.

Lineage validation is enforced in service/repository logic:
- recommendation runs can only reference runs in the same business/site
- recommendations can only be inserted for matching recommendation run lineage
